# Conformance Evidence Standard

- Standard ID: `cognitiveos.standard.conformance-evidence/0.1`
- Version: v0.1 Draft
- Machine version: `0.1.0-draft.1`
- Status: Draft Normative Standard
- Date: 2026-07-20
- Machine assets: `specs/schemas/profile-manifest.schema.json`,
  `conformance/vectors/*.json`
- Normative sources: `conformance/README.md`;
  `docs/evaluation/agent-benefit-benchmark.md` (REQ-PERF-005)

## 1. Scope and normative language

RFC 2119/8174 language applies. This standard fixes result-state vocabulary,
evidence rules, and manifest generation for the reference implementation's
conformance pipeline. Owning requirements: [REQ-CONF-001] through
[REQ-CONF-006], [REQ-PERF-002], [REQ-PERF-004], [REQ-PERF-005]. It registers
no new requirement.

## 2. Result states

Per-vector, per-requirement outcomes use exactly the four executed states of
`conformance/README.md`: `pass`, `fail`, `not-applicable`,
`documented-degradation`. Runner reports additionally use the report-level
state `not-run` for enumerated-but-not-executed vectors. Reconciliation of
the two vocabularies:

1. `not-run` is never an executed outcome and never enters a conformance
   claim; it exists so a report can honestly cover the whole vector set.
2. `pass` requires actual execution of the vector `input` against an
   implementation, comparison with `expected`, and preserved evidence.
   Parsing or schema-validating the vector file is NOT execution
   ("Do not report a vector as passed merely because the JSON parses").
3. `not-applicable` requires a recorded justification tied to declared
   scope; `documented-degradation` on an applicable MUST forces the profile
   claim down (shrink scope or `experimental`); safety negatives can never
   be degraded away.

## 3. Runner self-check

From M1, the runner suite MUST include at least one deliberately wrong
implementation (schema-valid outputs, wrong behavior) and demonstrate that
it fails the affected vectors. A runner release that cannot fail the wrong
implementation is itself non-conforming ([REQ-CONF-002] family; counters
F-015/F-016 recurrence).

## 4. Evidence rules

Evidence artifacts live under `artifacts/evidence/` (gitignored) and are
referenced by digest: a profile manifest `test_runs[]` entry pins
`suite_digest` and `result_ref`; performance claims pin `report_digest`
([REQ-PERF-004], `performance-report-contract.json`). Committed documents
cite evidence by path + digest, never by pasted content. Editing an emitted
evidence file invalidates its digest and thus every claim that pinned it.

## 5. Manifest generation

Profile manifests are generated by the runner, never hand-written. The M0
sample manifest declares every profile `planned`, zero `test_runs`, and
therefore no conformance claim; it exists to prove the manifest pipeline
validates against `specs/schemas/profile-manifest.schema.json`. A profile
moves to `implemented` only when every applicable MUST has passing evidence
or a justified `not-applicable` (four-state status language,
`conformance/README.md`).

## 6. Provisional M0 digests

Until the M1 bundle-manifest standard lands, the sample manifest pins:
`requirement_set_digest` = digest of the canonical JSON projection of
`specs/registry/requirements.yaml` under domain `spec-set/0.1`;
`schema_bundle_digest` = digest of a canonical manifest
`{"assets":[{id, content_digest}...]}` (sorted by id, per-schema canonical
content digests) under domain `schema-bundle/0.1`
(`canonical-encoding-and-digest.md` section 13). These are provisional
operational digests of the reference pipeline, not registered specification
set digests; M1 replaces them with the registered bundle procedure.

## 7. Agent-benefit claims

Any claim of agent benefit follows [REQ-PERF-005] and
`docs/evaluation/agent-benefit-benchmark.md`: four-arm design
(native / governance-only / optimized / ablation), preregistered thresholds,
and non-inferiority never reported as improvement. Governance overhead
reporting follows [REQ-PERF-004] and starts at M6 with a declared ungoverned
baseline.

## 8. Compliance checks

CI runs the enumerate-only runner and validates the emitted sample manifest
against the profile-manifest schema. From M1, CI additionally executes
layered vectors and the wrong-implementation self-check; `not-run` counts
must monotonically decrease per milestone plan.
